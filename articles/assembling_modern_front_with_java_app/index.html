Assembling modern front-end with Java app

Previously lots of Java powered web systems had their interface written through jsp files or some template engines like Freemarker or Thymeleaf. Using those starts to be painful when it comes to writing more complex applications, not even mentioning the SPA or PWA with them. Probably the strongest pain with those solutions is the need of full page reload to every requested URL path.
Then we were introduced to modern front-end frameworks, making the front-end development a little bit easier and natively creating the user web interface more performant.

The new struggle came with packaging the builds from those front-end frameworks to the Java web application binaries.

The simplest way

Such way could be done by running some commands through console, which would build and copy the front build files to some kind of webapp or static directory of the Java app, and then build the server app.
Using the console commands while working with Java is, from the definition, a bad idea, because it results in writing system specific scripts.
Whats more, such front-end frameworks relies on the NodeJS (and one of its package manager) for almost all of their needs.

Other way would be to keep all front-end resources and code in the root project directory or under the resources web of server app, but this would make our project structure harder to manage and understand. Imagine the mess and possible conflicts if the project would have separate front-end and back-end teams.

A "better" way

Nowadays, the usage of automation build tools for development of any kind of app is a standard practice.
Such tools are taking away the struggles with dependency management, cleaning/testing project and allows to build much more complex solutions with little configuration requirement.

Within the project configuration files of such tools, developing and packaging the front-end builds gets much simpler.

The examples provided in this article were made with the use of Gradle (6.0.1 version), Spring and Preact. It's possible to make similar configuration with Maven. To do so, you would need to use alike plugins and configure it properly.

It's recommended to use Gradle because it makes the configuration much smaller and allows to create some useful goals/tasks without using more additional plugins.
Check out the article about gradle which covers basics of Gradle features and migration from Maven.

The final look of working configuration will depend on your current project structure, specific build requirements and the java web frameworks being in use.
Aligning it to your needs shouldn't complicated, take a look at article linked above to see how you could amend the build phase accordingly to your needs.

Prerequisites

As mentioned, the Gradle, but some of IDEs like IntelliJ IDEA comes with bundled Gradle installation.
Few assumptions for provided examples in the next sections of this article are:

  • The project is made within multiple modules, for example having structure similar to:

    Root project 'j-root'
+--- Project ':j-web'
\--- Project ':j-server'
  • There are no sources in the root project
  • j-server module contains sources for Java application using Spring framework
  • j-web module is the root of front-end application source and uses Yarn as package manager
  • both modules are managed through Gradle (being included in settings script and having their own build.gradle files)

With all of those the structure of root project should be similar to (some files were omitted):

j-root
|   build.gradle
|   settings.gradle
+---assets
|   |   favicon.ico
|   \---icons //omitted the icon list to maintain readability
+---j-web
|   |    build.gradle
|   |    package.json
|   +---src
|   |       index.js
|   |       style.css
+---j-server
    |    build.gradle
    \---src
         \---main (having `java/pac.kage/Application.java` under it)

As mentioned earlier, changing the configuration to your project needs should take only a little bit of elbow grease.
The parts requiring amendment probably will be the project and module name, paths and task dependencies.
If you are new to Gradle, refer to the "Understanding Gradle" article.

Let's get started

Configuring the front-end module

To allow the usage of node and npm in Gradle tasks, the additional plugin is needed.
Add the below to the top of build.gradle in front-end module:

plugins {
    id "com.github.node-gradle.node" version "2.2.0"
}

This plugins allows for easy integration with NodeJS, Npm and Yarn tools.
Below the plugins block add:

node {
    version = '13.2.0' 
    yarnVersion = '1.19.2'
//  npmVersion = '2.1.5'  //this for npm
    download = true
}

This is the plugin configuration block, it determines the tool version used with the plugin.
Without having the download = true, the plugin will try to use globally installed NodeJS.

Next is to add gradle tasks which executes the front-end commands (to front module build script):

task build(type: YarnTask) {
    group = 'build'
    description = 'Build the client bundle'
    dependsOn('yarn')
    args = ['run', 'build']
}
task test(type: YarnTask) {
    group = 'verification'
    description = 'Run the client tests'
    dependsOn('yarn')
    args = ['run', 'test']
}

Kind of self descriptive, isn't it?
Similarly, you can add different tasks covering the run commands contained in your package.json file.
Now running the gradlew build from root project will execute the build command defined in package.json file.

To make npm task, the type: NpmTask should be used and script name changed accordingly to the one you have in package.json.

For more information take a look at the plugin documentation page.

Packaging to the server

With assumed project structure there is no need the change the build script of the server.
But if your root project contains the java web app sources, then you just need to change the task dependencies to refer to root project.

Add the following tasks to the build script of root project:

task copyClientResources {
    dependsOn(':j-web:build') //builds front-end at first
    doFirst {
        copy {
            from project(':j-web').buildDir.absolutePath
            into "${project(':j-server').buildDir}/resources/main/static"
        }
    }
}
task assembleServerAndClient {
    group = 'build' //shows task in 'build' group
    dependsOn(copyClientResources) //runs this task before itself
    finalizedBy(':j-server:build') //finalizes execution with provided task
    //both of the above could be replaced with single:
    //dependsOn(copyClientResources, ':j-server:build')
}

Now, running the assembleServerAndClient will build and copy front-end files to the static resources of server, which will be packed into binary generated by build task of server module.
If your root project is the Java app you can use ${rootProject.buildDir}/resources/main/static as the into copy path and refer to its build task with simple finalizedBy(build).

The copy paths may vary on your configuration. For example the Micronaut framework serves the static content from resources/main/public path and uses the ShadowJar plugin to build binaries with dependencies.

Few tips

Jumping through windows to start/stop the Java server or front-end development server, might be painful.
And it would be better to be able to manage it with Gradle tasks.
For starting the development of server such task can be added to build script of front-end module:

task startDevFront(type: YarnTask) {
    group = 'application'
    description = 'Run the client app'
    dependsOn('yarn')
    args = ['run', 'dev']
}

There's one little issue, stopping that gradle task won't stop the Node dev server.
To still be able to work through Gradle similar task can be added to front module:

task stopDevFront(type:Exec) {
    group = 'application'
    description = 'Stop devFront (kills all node.exe processes)'
    println 'Discovered Windows system'
    commandLine 'cmd', '/c', 'taskkill /F /IM node.exe'
}

The configured commandLine is for Windows OS, but this task can be simply extended to discover on which OS it's running and configure proper console command with simple if statements (see this SO question).

Running the java server with the newest build of front-end can be done with below task added to root build script:

task bootRunWithNewFront {
    group = 'application'
    dependsOn(copyFrontToServer)
    finalizedBy(':j-server:bootRun') //bootRun is the task created by Spring Boot 
}

Ending word

There are many ways of structuring such projects and packaging user web interface made with any modern JS framework to the server application, but many of them leads to overcomplicated structure and the need of making some tweaks while building one or another.
The presented approach of using Gradle and making those two a submodules, allows to create separate configurations based on their needs and keeps the project structure simple. Additionally, the use of shown plugin reduces the need of preinstalling the Node and Gradle tool makes the project and its configurations very minimal.
